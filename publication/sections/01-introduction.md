
## 1. Introduction

The emergence of Large Language Models (LLMs) as intelligent runtime environments has fundamentally altered the landscape of software architecture. Where traditional software systems require precise, imperative code to specify every execution step, LLM-based systems can interpret natural language descriptions of intent and autonomously determine how to achieve specified goals. This capability has given rise to a rapidly growing field: LLM-based Multi-Agent Systems (MAS), where multiple AI agents collaborate to accomplish complex tasks [1]. This paper focuses on a particular subset—Declarative MAS—where agents are defined through natural language blueprints rather than compiled code.

However, this nascent field lacks a shared vocabulary for design. Development teams building declarative MAS repeatedly encounter similar challenges—how to define agent behavior, how agents should communicate, how to ensure output quality, how to prevent AI hallucinations—yet each team develops ad hoc solutions without the benefit of accumulated collective wisdom. This situation mirrors the state of object-oriented programming before the publication of design patterns [2]: practitioners solving the same problems independently, unable to efficiently transfer hard-won insights.

This paper presents a Pattern-Oriented Multi-Agent System Architecture (POMASA)[^pomasa], a pattern language for constructing declarative multi-agent systems.

[^pomasa]: https://github.com/eXtremeProgramming-cn/pomasa POMASA is neither a framework (it imposes no mandatory code structure) nor a tool (the generator we provide is merely one way to apply it). At its core, POMASA is a pattern language [3]—a network of interconnected design patterns that guide both humans and AI in building MAS.

POMASA originated from building AI-powered research systems for social science—domains such as political economy, geopolitics, and international relations—where the system builders are domain researchers rather than software engineers. In these systems, no programming code is used; agents are defined entirely through structured markdown files and executed on intelligent runtimes such as Claude Code or Cursor. This context shaped a key design goal: the pattern language must be accessible to practitioners who think in terms of research methodology and domain knowledge, not in terms of APIs and data structures. At the same time, POMASA addresses general challenges in multi-agent coordination that arise regardless of domain, making it applicable to any team constructing declarative MAS. While the case study in this paper uses Claude Code as its intelligent runtime, the patterns themselves are runtime-agnostic; Section 8.2 discusses cross-platform portability and its current constraints.

The pattern language comprises 20 patterns organized into four categories: Core patterns (COR) that define fundamental system characteristics, Structure patterns (STR) that organize static system architecture, Behavior patterns (BHV) that define dynamic system behavior, and Quality patterns (QUA) that ensure system reliability. Each pattern is further classified by necessity: must patterns that every system requires, recommended patterns that most systems benefit from, and optional patterns for specific scenarios. Of the 20 patterns, 6 are must patterns that form the non-negotiable foundation of any declarative MAS. Together with 2 recommended patterns that enable multi-agent coordination, these 8 essential patterns form a self-contained network sufficient to build a working system—the focus of Section 5.

A distinctive property of this pattern language is that **for AI runtime environments, pattern languages themselves become executable**. Pattern languages have always faced a gap between description and implementation: practitioners must first learn the patterns, then figure out how to implement them—a barrier that has limited the reach of pattern knowledge. LLM-based intelligent runtimes bridge this gap. Because these runtimes can interpret natural language directly, a pattern language no longer needs to be read only by humans; it can be read by AI and used to generate working implementations. POMASA is designed with this property in mind, and we elaborate its implications in Section 7.

We demonstrate POMASA through an industry analysis system that employs 8 agents to conduct systematic research on industrial sectors, processing information through a pipeline of collection, verification, analysis, and synthesis to produce academic-quality reports with full data lineage.

The remainder of this paper proceeds as follows. Section 2 introduces the industry analysis system that serves as our running example. Section 3 surveys related work. Section 4 describes how the patterns were discovered and how they are organized. Section 5 presents eight essential patterns in detail. Section 6 demonstrates how patterns combine to form a working system. Section 7 elaborates the key insight about executable pattern languages. Section 8 discusses implications, limitations, and concludes.
